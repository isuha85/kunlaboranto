## DBA@OJT

Q: 초보 SI 프로젝트에서 DBA 에게 요구되는 상황 예
A:

+ 산출물
	+ WBS
	+ DB 사이징
	+ SQL작성가이드
		+ SQL 줄맞추기 
			+ TAB 사용 지양 ( 공백 4로 통일 )
		+ 모든 SQL의 선두위치에 / * SQL식별자 * / 주석 필요
		+ 힌트는 필요하면 사용해도 되나 PARALLEL 힌트를 추가시는 DBA와 협의 필수
		+ PK 및 UK 는 APP에서 결정하나, 성능을 위한 일반인덱스는 DBA가 주도하여 결정
	+ 테이블정의서
	+ 운영자가이드
	+ ERD
+ 통상의 요구들
	+ AS-IS 에서 TO-BE 환경으로 DB 구축
		+ 스키마 이관 
			+ 이종 DBMS 이고 객체수가 많은(수천) 경우 2일 소요 ( 정규식 지식 필요 )한다
		+ 데이타 이관
			+ 1억/10억 단위의 테이블인 경우 이행 시간 중요
			+ LOB 포함인 경우 천만건 단위도 이행 시간 유의
			+ 본이행을 생각한다면, 이행 중간시점에도, 남은 시간 예상이 가능해야함
			+ FK 가 있는 경우 NoValidate 후 Validate 가 필수라면, Validate 는 오래 걸리므로 후속 뒷단 작업으로 계획
	+ 개발단계에서의 일상적인 형상변경 요청
		+ 작업예
			+ PK 재생성
			+ 컬럼 추가 및 DEFAULT 값 설정
			+ 특정 위치에 컬럼 추가
			+ 컬럼 타입 및 길이 변경
			+ 속성 변경 ( NOT NULL -> NULL )
			+ 컬럼명 변경
			+ 테이블명 변경
		+ DBA 작업이, LOCK 등으로 방해되는 시간을 최소화 습관 필요
		+ 기본적으로 모든 DDL 권한은 일반계정에 부여하지 않음
		+ 예외적으로(개발중편의상) 아래는 허용도 가능
			+ PROCEDURE 변경을 위한 권한
			+ 특정 세션을 KILL 하는 권한
		
	+ DBA가 (비공식으로 보험상 해두는) 일상작업
		+ 테이블 단위의 백업 ( 특정시점데이타 조회기능이 있는 DB제품은 이를 ON 설정하고 길게 보존 )
		+ 자체 모니터링 설정 
			+ 과거를 다시 찾아볼 수 있어야하고, DBA가 필요한 부분을 계속 수정/추가 가능해야함
			+ ASHViewer / Grafana 형태의 차트 방식도 있으면 좋으나 필수는 아니며, TEXT 파일로 로그를 남겨야함
				+ LOCK 이력 (가해/피해 SQL 및 세션 정보포함)
				+ Active Session 정보 
				+ 그외 필요한 것
	+ 통합테스트 / 성능테스트 대응
		+ 통테전 또는 오픈직전이라도 대외테스트 전에, 모든 FULL SCAN 쿼리는 전수(?) 검사
			+ 일과중에 수행되는 것만
	+ 대외테스트 대응
	+ 공식 문서 작성
	+ DB관련 오픈 시나리오 준비 및 사전 리허설

Q: 초보 DBA 교육용 테이블 및 데이타 샘플 추천
A:

초보 DBA 교육을 위해서는 단순한 데이터 저장소를 넘어, **데이터베이스 관리의 핵심 요소(인덱스, 트랜잭션, 제약조건, 쿼리 튜닝 등)를 모두 경험할 수 있는 샘플**이 필요합니다.

DBA 입문자가 반드시 거쳐야 할 학습용 샘플 모델 3가지를 추천해 드립니다.

---

### 1. 전설적인 표준 샘플: AdventureWorks (SQL Server)

전 세계 DBA와 개발자들이 가장 많이 사용하는 Microsoft의 공식 샘플 데이터베이스입니다.

- **특징:**
    
    - **방대한 기능:** 단순히 데이터를 저장하는 것을 넘어, 파티셔닝, 인덱스(Clustered/Non-Clustered), XML 데이터 타입, 뷰(View), 저장 프로시저 등 거의 모든 SQL Server 기능을 포함하고 있습니다.
        
    - **현실성:** 실제 기업용 ERP 시스템과 유사한 복잡도를 가지고 있어, **실무 수준의 성능 튜닝 테스트**에 가장 적합합니다.
        
- **학습 포인트:** 대용량 데이터 환경에서의 인덱스 재구성(Rebuild/Reorganize), 통계 정보 관리.
    

### 2. 고전 입문용: Northwind

오래되었지만 가장 직관적인 모델입니다. 주문, 제품, 고객, 직원 간의 관계가 명확합니다.

- **특징:**
    
    - **단순함:** ERD(Entity Relationship Diagram)를 이해하고 테이블 간의 관계(Foreign Key)를 파악하는 기초 학습에 최적입니다.
        
    - **가벼움:** PC 자원을 거의 먹지 않아, 노트북 환경에서도 자유롭게 백업/복구 테스트를 수행할 수 있습니다.
        
- **학습 포인트:** JOIN 연습, 제약조건(Constraints) 설정, 데이터 정규화(Normalization).
    

### 3. DBA 실무형: E-commerce 커스텀 모델

DBA라면 스스로 테이블을 설계하고 데이터의 **카디널리티(Cardinality, 데이터의 분포)**를 직접 제어해봐야 합니다. 제가 추천하는 기본 구성은 다음과 같습니다.

- **테이블 구성:**
    
    - `Users`: 회원 정보 (Null 허용 컬럼, 고유 인덱스 연습)
        
    - `Products`: 상품 정보 (카테고리별 분류, 가격 범위 설정)
        
    - `Orders`: 주문 정보 (시간 순 데이터 분포 연습)
        
    - `Order_Items`: 주문 상세 (대량의 행이 생성되는 테이블, 인덱스 테스트의 핵심)
        
- **학습 포인트:**
    
    - **데이터 분산:** `Orders` 테이블에 10만 건 이상의 더미 데이터를 넣고 **인덱스 스캔 vs 시크** 성능 비교.
        
    - **성능 튜닝:** 특정 기간의 주문을 조회할 때 인덱스가 타는지 확인.
